<a id="そもそもスキルって何"></a>
## 1. そもそも「スキル」って何？

**スキル** は「Claude に渡しておく作業マニュアル」のようなものです。

たとえば「VB.NET の WinForms でフォームを作って」と頼むと、Claude は何も渡さなくてもコードを書けます。でも、そのコードが**あなたの現場の流儀**に合っているとは限りません。

そこで **「自分の現場ではこう書いてほしい」というお願い事を、あらかじめ紙に書いて渡しておく** のがスキルです。Claude は毎回この紙を参考にしてくれるので、指示を毎回繰り返さずに済みます。

このプラグインに入っている 5 つのスキルは、VB.NET WinForms の開発でよくある 5 種類の悩みごとに 1 枚ずつ用意した「お願い事の紙」のセットです。

## 2. 5 つのスキル、それぞれ一言で

| 名前 | 一言で言うと | いつ使う？ |
|---|---|---|
| **vb-project-setup** | プロジェクトの**土台**を整える人 | 新しいソフトを作り始めるとき |
| **vb-winforms** | **画面（フォーム）** の作法を教える人 | フォームを作る・直すとき |
| **vb-di-patterns** | 部品の**つなぎ方**を整理する人 | 「この機能をあちこちで使いたい」とき |
| **vb-concurrency-patterns** | **UI を固まらせない**方法を教える人 | 時間のかかる処理をするとき |
| **vb-coding-standards** | **書き方のお手本**を示す人 | どう書けば綺麗か迷うとき |

## 3. 困った場面 → どのスキル？（逆引き）

**「こんな場面で困った」を思い浮かべて、該当する行を探してください。**

### 新しく何かを始めるとき

| 困った場面 | このスキル |
|---|---|
| 新しい VB.NET のソフトを一から作りたい | `vb-project-setup` |
| `.sln` ファイルが古くさい、もっと見やすくしたい | `vb-project-setup` |
| NuGet パッケージのバージョンを 1 箇所でまとめたい | `vb-project-setup` |
| `.NET` のバージョンをどう指定すれば安心？ | `vb-project-setup` |

### フォーム（画面）が気持ちよく動かないとき

| 困った場面 | このスキル |
|---|---|
| フォームに処理が詰め込まれすぎて巨大になった | `vb-winforms` |
| `.Designer.vb` を触ったら動かなくなった | `vb-winforms` |
| フォームのコードをテストしたいけど、画面なしでテストできない | `vb-winforms` |
| 入力チェック（「数値だけ」「必須」など）をきれいに書きたい | `vb-winforms` |
| リストと詳細画面を連動させたい | `vb-winforms` |

### 部品のつなぎ方で迷ったとき

| 困った場面 | このスキル |
|---|---|
| 同じ処理をあちこちのフォームで使っている、まとめたい | `vb-di-patterns` |
| 子フォームを `New` で開くと、その中で何も動かない | `vb-di-patterns` |
| 「データベース接続」をいろんな場所で使いたい | `vb-di-patterns` |
| テスト用に「本物の代わりに偽物」を差し込みたい | `vb-di-patterns` |

### 「処理中 UI が固まる」問題のとき

| 困った場面 | このスキル |
|---|---|
| ボタンを押したら画面が固まって、何もできなくなる | `vb-concurrency-patterns` |
| 途中でキャンセルしたいのに、できない | `vb-concurrency-patterns` |
| 「進捗 30%」のように進み具合を見せたい | `vb-concurrency-patterns` |
| ファイル読み書きや通信でフリーズする | `vb-concurrency-patterns` |

### 書き方で迷ったとき

| 困った場面 | このスキル |
|---|---|
| C# のサンプルを VB に直したいが書き方がわからない | `vb-coding-standards` |
| 例外（Exception）と普通のエラー、どっちで返せばいい？ | `vb-coding-standards` |
| コードレビューの基準を整えたい | `vb-coding-standards` |
| チームに「この書き方で統一して」と示したい | `vb-coding-standards` |

<a id="各スキルをやさしく解説"></a>
## 4. 各スキルをやさしく解説

### 4-1. `vb-project-setup` — 土台づくりの人

#### 何をしてくれる？

**新築工事でいう「地盤と基礎」**を作るスキルです。

プログラムは「プロジェクト」という単位で作ります。プロジェクトには決まった書き方・並べ方があって、ここを適当にすると後で**必ず**困ります。このスキルは、そこを一発で整えてくれます。

#### 具体的に何が整う？

- **フォルダの並べ方**（どこに何を置くか）
- **パッケージ（部品）のバージョン管理**
  - 昔は各プロジェクトに「パッケージ A は v1.0 を使う」と書いていた
  - 今は**1 箇所にまとめて書く**（中央管理）のが推奨
  - このスキルはこれを自動でセットアップ
- **.NET のバージョン固定**（知らない間にバージョンが上がって動かなくなる事故を防ぐ）
- **ソリューションファイル**（`.sln` / `.slnx`）の正しい作り方

#### どんなとき呼び出される？

「新しくプロジェクトを作りたい」「このプロジェクトを整理したい」と Claude に伝えると自動で動きます。

---

### 4-2. `vb-winforms` — 画面の作法を教える人

#### 何をしてくれる？

**「フォームは器。ロジックは別に置く」**というお作法を教えるスキルです。

#### どうして別に置く？

多くの開発者が経験する、あの現象。最初は小さかったフォームが、処理を足していくうちに**何千行もの怪物**になっていく。

これはフォームが「画面」と「処理」を**両方背負ってしまった**結果です。このスキルは、この 2 つを分ける方法を教えます。

#### 例え話

**レストランで考えてみてください:**

- **ホール係（フォーム）** = お客さんの注文を聞く、料理を運ぶ
- **シェフ（Presenter・Service）** = 料理を作る、仕入れを決める

この役割分担があるから、レストランは回ります。もしホール係が厨房も切り盛りしたら、注文ミスと焦げた料理の嵐です。

プログラムでも同じで、**フォームは「聞く・見せる」だけ**、**実際の判断と処理は別の人（Presenter / Service）**、という分け方をします。このスキルはそれを徹底します。

#### 具体的に何を教えてくれる？

- フォームとロジックの分け方（MVP パターン）
- `.Designer.vb` を**触ってはいけない**理由
- リスト表示と詳細画面の連動（データバインディング）
- 入力チェックのきれいな書き方（`ErrorProvider`）
- 時間のかかる処理で UI を固まらせない方法

---

### 4-3. `vb-di-patterns` — 部品のつなぎ方を整理する人

#### DI って何？

DI = **Dependency Injection**（依存性の注入）。難しい名前ですが、やっていることはシンプルです。

#### 例え話で理解する

**電化製品のコンセントを思い浮かべてください。**

- 昔: 電化製品の中に電池が直接埋め込まれていた → 電池交換できない、壊れたら全取っ替え
- 今: コンセントに差す → 電源は外から来る、電化製品は電気があることだけ知っていればいい

プログラムもこれと同じで、**「部品を内部で作らず、外から渡してもらう」** のが DI です。

**何が嬉しい？**

1. **テストが楽になる** — 本物の代わりにテスト用の偽物を差し込める
2. **使い回しが楽になる** — 同じ「データベース接続」をあちこちで共有できる
3. **壊れにくくなる** — どこが何を使っているか一目でわかる

#### このスキルが教えてくれること

- `Program.vb` で最初に部品一覧をまとめて登録する方法
- 部品の「寿命」の決め方
  - Singleton（起動から終了まで 1 つ）
  - Scoped（1 つの作業の間だけ）
  - Transient（毎回新品）
- 子フォームを開くときに、部品を正しく渡す方法
  - **ここが落とし穴**:「子フォームを `New MyForm()` で作る」と、DI が効かなくなる！

### 4-4. `vb-concurrency-patterns` — UI を固まらせない人

#### 何を教えてくれる？

**「重い処理を動かしている間に、画面を固まらせない」**方法を教えるスキルです。

#### なぜ画面が固まるの？

WinForms は、**1 人の料理人（UI スレッド）**で動いています。この料理人は、以下の 2 つを全部一人でやります:

- 画面を描き直す
- ボタンが押された時の処理をする

ここに「ファイルを 10 秒読む」みたいな重い仕事を突っ込むと、料理人は**10 秒間それにかかりきり**になります。その間、画面を描き直せないので「固まった」ように見えるわけです。

#### どうすれば解決する？

**「重い仕事は別の人にやってもらう」** のが答えです。

VB.NET には `Async / Await` という仕組みがあって、「重い仕事は裏で進めつつ、料理人は他の仕事を続ける」が書けます。このスキルは、それを VB.NET 流に正しく書く方法を教えます。

#### このスキルの「三点セット」

1. **キャンセルできるようにする**（`CancellationToken`）
   - ユーザーが「やっぱりやめた」を押したら止まれるように
2. **進捗を見せる**（`IProgress(Of T)`）
   - 「30% 完了しました」みたいに、終わりが見える
3. **UI スレッドに戻る**（`Progress(Of T)`）
   - 裏で進めた結果を、ちゃんと画面に反映させる

#### ひとつだけ注意

**`Async Sub` は基本 NG**（使っていい例外的な場所が 1 つだけあるので、スキルに書いてあります）。
この 1 文だけ覚えておけば、重大バグの 9 割は防げます。

### 4-5. `vb-coding-standards` — 書き方のお手本の人

#### 何を教えてくれる？

**「VB.NET で綺麗に書くためのお作法集」**です。

#### どうして必要？

ネットで見かけるサンプルは、ほとんどが C# で書かれています。VB.NET に直そうとすると:

- **C# にあって VB.NET にない機能**がある（`record`、`init`、`with` 式 ...）
- **同じ機能でも書き方が違う**（`using` vs `Using`、`var` vs `Dim`）

そのたびに調べるのは大変。このスキルが**対応表と代替案**を持っているので、「C# のあれ、VB ではどう書く？」と聞けば答えが返ってきます。

#### このスキルが扱うトピック（ざっくり）

| トピック | 何の話？ |
|---|---|
| **値オブジェクト** | 「住所」「金額」などを**勝手に書き換えられないように**まとめる書き方 |
| **合成 > 継承** | 「親クラスを作って継承」より「部品を組み合わせる」方が扱いやすい |
| **Result パターン** | エラーを `Exception` で投げずに、**返り値で返す**書き方 |
| **パフォーマンス** | 大量データを扱うときのコツ（`Span`、`ArrayPool`） |
| **API 設計** | 他人が使いやすい関数の作り方 |
| **アンチパターン** | これはやめておけ、という典型 9 連発 |

#### 読み方のコツ

このスキルは他の 4 つと違って**「呼ばれて自動発火」しない**設計です。ユーザーが自分で「`/vb-coding-standards`」と明示的に呼ぶ、または「コーディング規約のこと」と言及したときだけ動きます。

理由: ちょっと VB を書くたびに発火されると邪魔なので、**参照したいときだけ**開く辞書にしています。

<a id="5-つのスキルは家を建てる仕事に似ています"></a>
## 5. 5 つのスキルは、家を建てる仕事に似ています

全体を家作りに例えると、こんな関係です:

```
        ┌──────────────────────────────────┐
        │  vb-coding-standards             │
        │  （大工の道具・技法の辞書）       │ ← 迷ったら開く
        └──────────────────────────────────┘
                      ↑ 参照
┌────────────────────────────────────────┐
│  vb-concurrency-patterns                │
│  （電気・水道が止まらない工夫）           │
├────────────────────────────────────────┤
│  vb-winforms      │  vb-di-patterns    │
│ （間取りと内装） │ （配管のつなぎ方）   │
├────────────────────────────────────────┤
│  vb-project-setup                       │
│  （地盤と基礎工事）                      │
└────────────────────────────────────────┘
```

**下が基礎、上に行くほど細かい工事。**
家は下から建てていくのが正しい順番なので、スキルもこの順で覚えると身につきやすいです。

<a id="使うときのコツ"></a>
## 6. 使うときのコツ

### コツ 1: 最初は 1 つのスキルに集中

5 つ全部を一気に覚えようとすると消化不良になります。**今やっている作業に関係する 1 つだけ**を開いてみるのが良いです。

例:
- 新規プロジェクトなら `vb-project-setup` だけ
- フォームを直しているなら `vb-winforms` だけ

### コツ 2: 「なんかおかしいな」のときが出番

スキルは**困ったときの辞書**です。以下のようなサインが出たら、そのテーマのスキルを開いてください:

| サイン | 開くスキル |
|---|---|
| フォームが 500 行を超えた | `vb-winforms` |
| 同じ処理を 3 箇所にコピペした | `vb-di-patterns` |
| 「処理中」で画面が固まる | `vb-concurrency-patterns` |
| C# のサンプルを見て「これ VB だとどうするんだろう」と思った | `vb-coding-standards` |
| `.sln` を開くたびに「この構成でいいのかな」と不安 | `vb-project-setup` |

### コツ 3: Claude に「このスキルで」と明示できる

ふつうは Claude が自動で適切なスキルを選びますが、**自分で指定することもできます**:

```
あなた「このコード、vb-winforms の観点でレビューしてください」
```

こう言うと、そのスキルの基準に絞ってチェックしてくれます。

### コツ 4: スキルは「完璧」を要求していない

スキルに書いてある通りに書けないとダメ、ということはありません。現場には現場の事情があります。
**まずは趣旨を理解**して、**現実的に守れる範囲で守る**のが健全です。
